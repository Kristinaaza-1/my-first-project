# AI-помощник-администратор для косметологов — спецификация MVP

## Executive Summary
Первый модуль модульной AI-платформы для косметологов: AI-помощник, который забирает у мастера рутинную переписку с клиентами (цена, адрес, что взять с собой, запись на приём) в Telegram/WhatsApp/Instagram и сам предлагает и бронирует время в уже существующем расписании мастера, передавая сложные и чувствительные вопросы напрямую мастеру. Цель — не продать ИИ, а вернуть мастеру время и снять хаос переписки в нескольких каналах одновременно.

## Problem Statement
Мастера ведут переписку с клиентами одновременно в нескольких каналах (WhatsApp, Instagram, Telegram, звонки), и, по признанию самих мастеров, почти вся эта переписка — рутина: цена, адрес, что взять с собой, когда можно записаться. При этом реакция на задержку ответа непредсказуема: где-то клиент уходит к другому мастеру, где-то просто ждёт. Мастер тратит на это личное время, которое могло бы пойти на работу с клиентами или на себя.

Из четырёх исходных гипотез («переписка», «контент», «поиск клиентов», «работа в нескольких сервисах») в ходе интервью выяснилось, что:
- «Поиск клиентов» — самая частая жалоба, но по сути это проблема рекламного бюджета/продвижения, а не софта — сама Кристина не уверена, что ИИ здесь решает задачу напрямую.
- «Единое окно сервисов» на практике означает «переписка + запись на приём в одном месте» — то есть по сути то же самое, что «переписка с клиентами», просто с другой стороны. Именно эта комбинация вызвала мгновенное «да, мне это нужно» при проверке ценности.

Почему сейчас: у Кристины уже есть отдельные контакты среди косметологов для проверки гипотезы, и это единственная из четырёх проблем, которая не зависит от внешних факторов (рекламного бюджета) и полностью решается силами продукта.

## Success Criteria
- Качественный сигнал: мастера в пилоте сами говорят «буду платить» без наводящих вопросов — это выбранная владелицей продукта метрика готовности двигаться дальше.
- Функциональный ориентир (не строгое обязательство на MVP, скорее рамка для оценки): заметная доля рутинных сообщений (запись, цена, адрес, подготовка к процедуре) закрывается без участия мастера.
- Экономия времени мастера на переписке — ощутимая лично для него, не обязательно измеряется точной метрикой на пилоте.

## User Personas
- **Основной пользователь MVP:** соло-мастер или фрилансер в бьюти-сфере, ведёт переписку в нескольких каналах сам, сам себе администратор и маркетолог. Не технический пользователь.
- **Будущие сегменты (не в MVP):** небольшие салоны/студии с администратором, клиники с CRM — потребуют ролей доступа и других сценариев, сознательно вынесены за скобки первого модуля.
- **Косвенный пользователь:** клиент мастера, который пишет в WhatsApp/Instagram/Telegram и either получает мгновенный ответ от помощника, либо передаётся мастеру напрямую.

## User Journey
1. Клиент пишет мастеру в любом из подключённых каналов (Telegram сразу; WhatsApp/Instagram — после одобрения Meta).
2. Помощник читает сообщение:
   - Если это рутинный вопрос (цена, адрес, что взять с собой, желание записаться) — отвечает сам и предлагает свободное время из расписания мастера.
   - Если это чувствительная или сложная тема (здоровье, аллергии, противопоказания, беременность, нестандартный случай) — **не обрабатывает и не запоминает содержание**, сразу передаёт мастеру с пометкой «нужен личный ответ».
3. Если клиент подтверждает время записи — помощник резервирует слот и отправляет мастеру предложение записи на подтверждение одним кликом.
4. Мастер одним действием подтверждает (или отклоняет/переносит) — запись фиксируется в подключённом календаре (Google Календарь или YCLIENTS/Altegio).
5. Мастер в течение дня открывает список дел/эскалаций, а не пять чатов параллельно.

## Functional Requirements

### Must Have (P0)
- Приём и разбор сообщений из Telegram (доступен сразу, без внешнего одобрения).
- Автоответы на рутинные вопросы: цена, адрес, что взять с собой, доступное время записи.
- Определение и немедленная эскалация чувствительных/сложных сообщений мастеру без сохранения их содержания ботом.
- Предложение слота записи и запись в расписание мастера через интеграцию с Google Календарём **или** YCLIENTS/Altegio.
- Подтверждение записи мастером одним кликом (запись не финализируется автоматически без подтверждения).
- Экспорт всей накопленной клиентской базы и истории переписки мастером в любой момент, включая уход с платформы.
- Доступ к переписке/данным клиентов в пилоте — только у самого мастера (без ролей команды/администратора).

### Should Have (P1)
- Подключение WhatsApp и Instagram Direct (требует прохождения Meta App Review — параллельный трек, не блокирует запуск на Telegram).
- Базовый учёт истории клиента (что делал, когда), кроме данных о здоровье.

### Nice to Have (P2)
- Поддержка DIKIDI и других сервисов записи.
- Роли доступа для администратора салона/нескольких мастеров.
- Тарификация по использованию (число диалогов/записей) как альтернатива фиксированной подписке.
- Напоминания клиентам о записи вне 24-часового бесплатного окна мессенджеров (это уже платные исходящие сообщения по тарифам WhatsApp/Instagram).

## Technical Architecture

### Data Model (укрупнённо)
- **Мастер**: профиль, подключённые каналы, подключённое расписание.
- **Клиент**: контакт, история обращений (без чувствительных данных о здоровье).
- **Диалог/сообщение**: канал, статус (закрыт ботом / эскалирован мастеру).
- **Запись на приём**: время, статус подтверждения, связанный клиент и мастер.

### System Components
- Канальные интеграции: Telegram Bot API (без ограничений на старте); WhatsApp Business API и Instagram Graph API (требуют прохождения Meta App Review, есть риск отказа с первой попытки).
- ИИ-логика ответов и определения эскалации (маршрутизация «рутина / чувствительная тема»).
- Интеграция с расписанием: Google Calendar API (бесплатно, щедрая квота) и/или YCLIENTS/Altegio REST API (официальный партнёрский API, доминирующая система записи в бьюти-сегменте СНГ).
- Реализация: no-code стек (например, Make/n8n + Airtable/аналог + AI API), собирает сама Кристина, первый опыт — потребуется сопровождение по ходу сборки.

### Integrations
- Telegram — без approval, доступен сразу.
- WhatsApp Business API — ответы клиенту в течение 24 часов после его сообщения бесплатны; исходящие сообщения вне этого окна платные (~$0.004–$0.05 за сообщение в зависимости от категории).
- Instagram Direct (Meta Graph API) — API бесплатный, но нужен App Review (дни-недели, частые отказы с первой попытки); лимит ~200 автосообщений в час на аккаунт — не критично для соло-мастера.
- Google Calendar API — бесплатно в рамках обычной нагрузки.
- YCLIENTS/Altegio — официальный партнёрский REST API, есть документация и поддержка для разработчиков.
- DIKIDI — вне MVP: публичный API неоднозначен, только сторонние no-code коннекторы; добавить позже при реальном спросе.

### Security & Access Model
- Доступ к переписке/данным клиентов на пилоте — только у мастера, без общих ролей.
- Данные клиентов принадлежат мастеру, не платформе; экспорт доступен всегда.
- Чувствительные данные о здоровье не читаются и не сохраняются помощником — только эскалируются человеку.
- Аудитория смешанная/международная — при выборе хранения данных ориентироваться на наиболее строгий применимый стандарт (в духе GDPR/152-ФЗ), юрисдикция окончательно не определена и требует уточнения при масштабировании за пилот.

## Non-Functional Requirements
- Скорость запуска важнее полноты: целевой срок первого прототипа — 2-4 недели, но это оптимистично при первом опыте с no-code и с учётом непредсказуемых сроков Meta App Review — реалистичный риск сдвига на 1-2 месяца для WhatsApp/Instagram веток, застрахован параллельным запуском на Telegram.
- Надёжность эскалации важнее автоматизации: при любой неуверенности бот должен передавать диалог мастеру, а не гадать — особенно в вопросах здоровья.

## Out of Scope (для этой версии)
- Решение проблемы «поиск клиентов» (реклама/лидогенерация) — вне MVP, требует отдельного исследования, зависит от бюджета клиента, а не только от продукта.
- Модуль генерации контента.
- Роли команды/администратора салона, мульти-мастерские кабинеты, клиники и CRM-интеграции корпоративного уровня.
- DIKIDI и прочие сервисы записи, кроме Google Calendar и YCLIENTS/Altegio.
- Оплата по использованию — на старте фиксированная подписка после бесплатного пилота.

## Open Questions for Implementation
- Итоговая юрисдикция и требования к хранению данных при выходе за пилотную аудиторию (смешанная сейчас, нужно определить приоритетный рынок).
- Точный размер и состав пилотной группы мастеров — пока не определён («сколько наберётся»), стоит определить по мере привлечения контактов.
- Точная цена подписки после пилота — сознательно не зафиксирована, определится после первых сигналов готовности платить.
- Что происходит с сообщением, если мастер не отвечает долго на эскалацию/подтверждение записи (нет чёткого паттерна ожидания у разных мастеров) — нужно решить поведение бота в этом случае (например, напоминание клиенту, что мастер скоро ответит).

## Appendix: Research Findings
- WhatsApp Business API: с 1 июля 2025 модель оплаты — по отдельным сообщениям, а не по «диалогам»; ответы клиенту в течение 24 часов бесплатны («service messages»); исходящие вне окна — $0.025–$0.1365 (маркетинг) или $0.004–$0.0456 (утилитарные/подтверждения).
- Instagram Graph API: API сам по себе бесплатный; нужен Meta App Review (дни-недели, частые отказы на permissions вроде `instagram_manage_messages`); лимит 200 автосообщений/час на аккаунт.
- Google Calendar API: бесплатен при обычной нагрузке, лимиты по умолчанию щедрые (в терминах запросов в день/минуту).
- YCLIENTS/Altegio: официальный REST API для партнёров-разработчиков, есть документация, партнёрская поддержка; Altegio — международный бренд той же компании, что и YCLIENTS.
- DIKIDI: информация о публичном API противоречива — часть источников утверждает, что официального API нет, интеграции идут через сторонние no-code коннекторы (Albato, ApiMonster).
